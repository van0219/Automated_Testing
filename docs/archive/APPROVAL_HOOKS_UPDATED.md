## Updated Hooks

### 1. Approval Step 1: Parse TES-070 Document (v2.0.0)

**Purpose**: Parse existing TES-070 documents and generate framework-compatible test scenarios

**Changes**:
- ✅ Now generates framework-compatible JSON format
- ✅ Creates standard 7-step workflow for each scenario

**Output Format**:
```json

  "description": "...",
  "scenarios": [
    {
      "scenario_id": "3.1",

        {"number": 2, "description": "Navigate to Payables", "action": {...}},
        {"number": 3, "description": "Create transaction", "action": {...}},
        {"number": 4, "description": "Submit for approval", "action": {...}},
        {"number": 5, "description": "Navigate to Work Units", "action": {...}},
        {"number": 6, "description": "Wait for completion", "action": {...}},
        {"number": 7, "description": "Verify status", "action": {...}}

    }
```

**Standard 7-Step Pattern**:
1. Login to FSM
2. Navigate to module (Payables/GL/Cash)
3. Create transaction with scenario-specific data
4. Submit for approval
5. Navigate to Work Units
6. Wait for workflow completion
7. Verify final status

### 2. Approval Step 2: Execute Approval Tests (v2.0.0)

**Purpose**: Execute approval tests using the automated framework

**Major Changes**:
- ❌ Removed: Manual Playwright MCP calls (100+ lines of detailed instructions)
- ✅ Added: Framework execution code (simple Python snippet)
- ✅ Automatic evidence collection
- ✅ Automatic TES-070 generation
- ✅ Token efficient execution

**Old Approach** (Manual):
```
For each scenario:
  A. SETUP - Create directories, generate IDs
  B. LOGIN TO FSM - Manual Playwright calls
  C. CREATE TRANSACTION - Manual form filling
  D. SUBMIT FOR APPROVAL - Manual button clicks
  E. MONITOR WORK UNIT - Manual navigation and polling
  F. APPROVAL ACTIONS - Manual approver login and actions
  G. FINAL VALIDATION - Manual status checks
  H. RECORD RESULTS - Manual result tracking
```
**Result**: Consumed thousands of tokens per scenario

**New Approach** (Automated):
```python
orchestrator = TestOrchestrator(
    client_name='{ClientName}',
    environment='ACUITY_TST',
    logger=logger
)
result = orchestrator.run('Projects/{ClientName}/TestScripts/approval/{scenario_file}')
```
**Result**: Framework automates everything, minimal token consumption

**What Framework Does Automatically**:
- Initializes unique run_group identifier
- Loads credentials from .env files
- Executes all scenarios sequentially
- Captures screenshots at each step
- Monitors work units with adaptive polling
- Verifies final status
- Generates TES-070 document
- Cleans up resources

## Framework Architecture

### Key Components Used

1. **TestOrchestrator** - Main entry point
   - Initializes all components
   - Loads credentials
   - Executes scenarios
   - Generates TES-070

2. **ApprovalExecutor** - Approval-specific logic
   - Executes approval workflow steps
   - Handles FSM actions
   - Tracks results

3. **FSM Actions** - Modular action handlers
   - fsm_login - Login to FSM
   - fsm_payables - Payables operations
   - fsm_workunits - Work unit monitoring

4. **TestState** - State management
   - Tracks variables (run_group, work_unit_id, etc.)
   - Interpolates {{state.variable}} in configs
   - Maintains execution history

5. **ScreenshotManager** - Evidence collection
   - Captures screenshots automatically
   - Saves with proper naming
   - Organizes by scenario

6. **TES070Generator** - Document generation
   - Creates Word document
   - Inserts screenshots
   - Formats professionally

## Technical Notes

### Framework Execution Context
- Framework MUST run in Kiro's execution context
- Playwright MCP tools only available in Kiro context
- Cannot run as subprocess (`python run_tests.py` will fail)
- Kiro executes framework code directly

### State Interpolation
- Uses {{state.variable}} syntax
- Framework replaces at runtime
- Common variables:
  - {{state.run_group}} - Unique test identifier
  - {{state.password}} - FSM password from .env
  - {{state.work_unit_id}} - Work unit ID from execution

### Credential Management
- Loaded from Projects/{ClientName}/Credentials/
- .env.fsm - FSM URL and username
- .env.passwords - FSM password
- Never logged or exposed
